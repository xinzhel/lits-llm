# Output Files Reference

This document describes all files generated by `lits-search` and `lits-chain`.

## Directory Structure

### lits-search (tree search: MCTS, BFS)

```
{root_dir}/{model_prefix}_results/{dataset}_{method}/run_{version}/
├── checkpoints/                         # Incremental rollout paths
│   ├── {query_idx}_{iter}.json          # Path from iteration {iter}
│   └── {query_idx}_result.json          # Final selected path
├── terminal_nodes/                      # Terminal nodes for evaluation
│   └── terminal_nodes_{query_idx}.json
├── treetojsonl.jsonl                    # Complete search trees (all paths)
├── treetojsonl_unselected_simulate.jsonl  # Unselected simulate paths (MCTS only)
├── config.json                          # Experiment configuration
├── execution.log                        # Execution logs
└── inferencelogger.log                  # LLM call metrics (JSON lines)
```

### lits-chain (chain agents: ReAct, NativeReAct)

```
{root_dir}/{model_prefix}_results/{dataset}_chain/run_{version}/
├── checkpoints/
│   ├── {idx}.json                       # Single-attempt checkpoint (pass@1)
│   ├── {idx}_a{attempt}.json            # Per-attempt checkpoint (pass@N)
│   ├── {idx}_reward.json                # Verification result (if verify_fn exists)
│   ├── {idx}_a{attempt}_reward.json     # Per-attempt verification (pass@N)
│   └── pass_at_n_summary.json           # pass@N aggregate results
├── config.json
├── execution.log
└── inferencelogger.log
```

## File Descriptions

### config.json

Complete experiment configuration for reproducibility.
Saved by both `lits-search` and `lits-chain` after agent creation.

### terminal_nodes/ (lits-search only)

`terminal_nodes_{query_idx}.json` — all terminal nodes for post-search evaluation.
Used by `lits-eval` for answer extraction and scoring.

```json
{
  "terminal_nodes": [
    {"id": 5, "state": {...}, "action": "...", "reward": 0.8},
    {"id": 12, "state": {...}, "action": "...", "reward": 0.6}
  ],
  "query": "Original question text",
  "query_idx": 0
}
```

### checkpoints/

**lits-search**: `{query_idx}_{iter}.json` — serialized path (root to leaf) per MCTS iteration. `{query_idx}_result.json` — final selected path. Used for resume and real-time monitoring.

**lits-chain**: `{idx}.json` or `{idx}_a{attempt}.json` — serialized `ToolUseState` (full conversation history). `{idx}_reward.json` — verification result from environment-based benchmarks (e.g., Terminal-Bench). `pass_at_n_summary.json` — aggregate pass@1 and pass@N rates.

### treetojsonl.jsonl (lits-search only)

JSON Lines — one line per example, each containing all explored paths.
Used for tree visualization (`lits.visualize.visualize_tree()`).

### execution.log

Detailed execution logs. Log levels: DEBUG (step-by-step), INFO (progress), WARNING (issues), ERROR (failures).

### inferencelogger.log

JSON Lines — one record per LLM call. Tracks token usage, latency, and cost across all model roles (policy, reward, transition, memory).

```json
{"timestamp": "04-22 20:32:46", "role": "policy_0", "input_tokens": 1750, "output_tokens": 154, "batch": false, "batch_size": 1, "num_flatten_calls": 0, "running_time": 0.0, "cached": false, "attempt": 0}
```

Fields:
- `role`: `{component}_{query_idx}` (e.g., `policy_0`, `reward_3`, `memory_0`)
- `input_tokens`, `output_tokens`: token counts for this call
- `attempt`: pass@N attempt index (only present when `n_attempts > 1`)
- `batch`, `batch_size`: whether this was a batched call
- `cached`: whether the response was served from cache

## Result Directory Management

### Default naming

Both `lits-search` and `lits-chain` generate result directories as:
```
{root_dir}/{model_prefix}_results/{dataset}_{method}/run_{version}
```
Example: `results/us.anthropic.claude-3-5-haiku-20241022-v1:0_results/terminal_bench_chain/run_v0.3.2`

The `model_prefix` is derived from the policy model name (last segment after `/`).
The `version` comes from `pyproject.toml`.

### When to rename before re-running

Different experiment configurations that use the same model, dataset, and package version write to the **same** result directory. Without `--override`, resume logic skips completed checkpoints, but logs are appended to, mixing data from different configurations.

**Rename the existing run dir before re-running** whenever:
- Changing `n_attempts` (e.g., pass@1 → pass@5)
- Adding or removing `--memory-arg`
- Changing `max_steps`, `temperature`, or other config that affects results
- Any configuration change where you want to preserve the old results for comparison

**Naming convention**: append a descriptive suffix:
```
run_v0.3.2_pass1_pilot10    # pass@1, first 10 tasks
run_v0.3.2_pass5_pilot10    # pass@5, first 10 tasks
run_v0.3.2                  # current active run (no suffix)
```

The active run always uses the bare `run_{version}` name. Archived runs get a descriptive suffix.

## Evaluation Commands

```bash
# Evaluate lits-search results
lits-eval --result_dir {result_dir} --dataset_name {dataset} --include {module}

# Evaluate lits-chain results (env-grounded)
lits-eval-chain --result_dir {result_dir}

# Evaluate lits-chain results (tool-use with verification)
# Verification happens inline during lits-chain execution (reward files in checkpoints/)
```
